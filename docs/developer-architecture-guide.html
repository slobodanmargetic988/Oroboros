Ouroboros

Developer Architecture Guide

This document explains how Ouroboros actually works in code: run lifecycle, worktree isolation, failure recovery, test coverage, cross-stack feature changes, and system safety guardrails.

1) What The System Is

Ouroboros is a host-deployed control plane that accepts a change request, executes the change in a slot-bound git worktree, validates it, exposes preview status in the UI, and only merges after approval.

Frontend

/codex inbox, run details, approvals, notifications, route badges, quick prompt overlay.

Backend API

Run CRUD, lifecycle transitions, approvals/rejections, events, slot/worktree APIs, merge gate.

Worker

Claims queued runs, acquires slot lease, assigns worktree, executes Codex command, runs checks.

Infra Runtime

Host services with systemd, reverse proxy, release/rollback scripts, stale lease maintenance timers.

2) End-To-End Run Flow

  1. User creates run via POST /api/runs with prompt + context.
  2. Run starts at queued.
  3. Worker claims oldest queued run using DB lock (with_for_update(skip_locked=True)).
  4. Worker acquires one fixed slot lease (preview-1..3), moves run to planning.
  5. Worker assigns slot worktree and branch codex/run-<run_id>.
  6. Worker runs Codex command inside that worktree only, then required checks.
  7. On success run transitions to preview_ready; on failure to failed with reason code.
  8. Reviewer approves/rejects; approval triggers merge gate checks + merge path.

3) How Worktrees Are Used

Slot model

  • Exactly 3 preview slots by default: preview-1, preview-2, preview-3.
  • Lease has TTL and heartbeat; stale leases are expired/reaped.
  • If all slots are leased, run stays queued with WAITING_FOR_SLOT.

Branch and path binding

  • Each run branch is deterministic: codex/run-<run_id>.
  • Each slot has one worktree path under configured root (default /srv/oroboros/worktrees).
  • Binding is persisted (slot_worktree_bindings) and tracked in events/audit logs.

This gives isolation between parallel runs: no direct edits on main during run execution.

4) Recovery From Bad States

Failure Mode What Happens Recovery Path
Worker timeout Run transitions to failed with AGENT_TIMEOUT. /api/runs/{run_id}/resume creates child run when reason is recoverable.
Lease expired Heartbeat rejects lease, run can move to failed/expired path, slot released. Lease reaper + resume/retry flow.
Checks failed Run moves to failed with CHECKS_FAILED and failed check metadata. Fix code, then retry child run.
Canceled run Cancel endpoint transitions to canceled, attempts cleanup + lease release. Submit new run if needed.
Deploy health fails Deploy script supports automatic symlink rollback to previous release. scripts/rollback.sh <release_id> and health gate rerun.

5) What Tests Cover Right Now

Layer Covered Not Fully Covered Yet
Backend API Run resilience (cancel/retry/resume), approval + merge gate behavior, event stream/schema, metrics. Deeper end-to-end deploy orchestration on real host, broader route-level integration tests.
Worker Claim path, slot/worktree assignment behavior, validation pipeline outcomes, timeout/cancel semantics, guardrails. Large-scale concurrency soak tests, full real-world Codex command behavior under load.
Frontend Utility logic (status chips, diff parsing, failure reason extraction, route helpers). Component-level/UI automation tests for full user flows.

Default required checks in worker/merge gate are currently lint,test,smoke and are configurable via environment variables.

6) How It Adds New Features Across Frontend + Backend

  1. Create one run with a prompt that describes both API/data changes and UI behavior.
  2. Agent edits both stacks inside one run worktree branch.
  3. Validation checks run; reviewer inspects run details (diff/checks/events).
  4. Approval triggers final checks pinned to expected commit SHA, then merge.

Because the repo is a monorepo, one run can include backend + frontend + worker updates in one coherent commit lineage.

7) Can It Edit Other Apps On The Same Server?

By default: effectively no, by design intent.

  • Worker executes in configured worktree root (WORKER_ALLOWED_PATHS default /srv/oroboros/worktrees).
  • Worker command allowlist blocks arbitrary executables and blocks shell wrappers explicitly.
  • Systemd hardening constrains writable paths to worktrees/artifacts/tmp for the worker service user.

Important caveat: this is strong host hardening, not a full VM/container sandbox.

  • If allowlists are loosened too much, blast radius increases.
  • If you run worker manually outside systemd hardening, protections are weaker.

8) Guardrails That Protect The Server

  • Command allowlist: blocks non-approved executables; disallows bash/sh/zsh wrappers.
  • Path allowlist: enforces worker cwd under approved roots.
  • Env isolation: preview subprocess inherits allowlisted vars and strips sensitive secret keys.
  • Least privilege OS user: worker runs as dedicated low-privilege account via systemd.
  • State machine enforcement: invalid status transitions are rejected server-side.
  • Merge gate commit pin: merge checks validate exact expected commit SHA to avoid drift.
  • Audit trail: run events + audit logs + structured traces for forensic visibility.

9) Current Practical Limits

  • Max active previews are tied to slot count (default 3).
  • Deploy lifecycle is represented in run status/events; host deploy/rollback scripts exist separately.
  • Frontend checks are not as deep by default as backend/worker checks unless you extend required checks.
  • No container isolation; security posture depends on strict host config and disciplined operations.

Recommended next hardening step: add frontend integration tests to required checks and run health-gated deploy script as part of approved merge workflow.
