Ouroboros

User Guide

Ouroboros is an AI change-request workspace for your app. You describe a change in plain language, the system builds it in an isolated preview, and you decide whether to merge it into production.

What it does

Turns natural-language requests into tracked engineering runs with preview links, checks, and approval control.

Who uses it

Product owners, developers, and reviewers who want safe AI-assisted changes without editing main directly.

How it stays safe

Every run has statuses, audit events, validation checks, and explicit review before a merge is allowed.

How The App Works In Plain Terms

  1. You open /codex and submit a prompt describing the change.
  2. Ouroboros creates a run and puts it in a queue.
  3. A worker builds the change in an isolated preview slot (not on production).
  4. Checks run (tests, health, and validation).
  5. You open preview, verify behavior, and approve or reject.
  6. If approved, the system merges to main and deploys.

Runtime model for this project: host-native processes (no Docker/containers required).

What You See In The UI

  1. Create Run: Prompt box with optional route context, note, and metadata JSON.
  2. Runs Inbox: Live list of runs, filters, slot occupancy, and preview links.
  3. Run Details: Diff, checks summary, timeline events, artifacts, and approve/reject controls.
  4. Route Badge: Small panel showing runs related to your current page.
  5. Notifications: Alerts for key outcomes like preview_ready, failed, and merged.
  6. Quick Prompt: Press Cmd/Ctrl + K to open prompt overlay from any page.

Status Meanings

queued planning editing testing preview_ready needs_approval approved merging deploying merged failed canceled expired

Typical happy path: queued -> planning -> editing -> testing -> preview_ready -> needs_approval -> approved -> merging -> deploying -> merged.

First-Time User Checklist

  1. Open /codex.
  2. Submit a small safe prompt first (for example, text/UI copy change).
  3. Wait until run reaches preview_ready or needs_approval.
  4. Open preview and test exactly what you requested.
  5. Review run details (diff + checks) before deciding.
  6. Approve only when behavior matches your expectation.

Troubleshooting Quick Notes

  1. If a run is stuck in queued, preview slots are likely full. Wait for a free slot.
  2. If a run is failed, open run details and inspect failure reason + check artifacts.
  3. If preview does not reflect new changes, refresh slot/inbox and verify you opened the correct run preview link.
  4. If backend or worker is offline, run states will not advance until services recover.
